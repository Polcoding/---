# Codex 작업 규칙

## 0. 이 파일의 역할

이 파일은 Codex가 이 저장소에서 작업할 때 반드시 따라야 하는 프로젝트 공통 지침입니다.

이 저장소는 공공기관 행정문서 AI 자동화 시스템을 단계적으로 설계ㆍ문서화ㆍ검증하는 프로젝트입니다.

이 프로젝트의 핵심은 AI가 업무를 직접 처리하는 시스템이 아니라, 비식별 업무 지시를 받아 공공기관 행정문서 초안을 생성하고 사람이 최종 검토ㆍ승인하는 시스템입니다.

현재까지 작업 범위:
- Custom GPT Instructions 설계
- Custom GPT 수동 테스트
- 보안등급 분류 기준 작성
- 비식별 샘플 작성 기준 정리
- 문서 패턴 라이브러리 작성
- HWPX/XLSX 템플릿 PoC 설계
- AI 출력 JSON 스키마 설계
- 템플릿 렌더러 요구사항 설계
- XLSX, Markdown, Email, HWPX 로컬 PoC 렌더러 작성

- 결과보고서와 검토보고서 HWPX 로컬 placeholder 템플릿 치환 테스트
- 핵심 HWPX 보고서 4종 로컬 placeholder 치환 및 수동 검수 완료
- 실제 HWPX 양식 투입 전 안전 절차와 공통 placeholder 설계 정리
- Phase 1 완료 기준과 Phase 2 진입 조건 정리

## 1. 최우선 원칙

- 실제 개인정보, 민감정보, 내부자료, 공공기관 원문자료를 생성하거나 저장소에 추가하지 않습니다.
- 기존 공문 원문을 AI에 학습시키거나 Custom GPT Knowledge에 바로 업로드하도록 안내하지 않습니다.
- 실제 자동화 구현보다 안전한 문서화, 테스트 기준, 검수 체계 정리를 우선합니다.
- 모든 산출물은 초안 생성 보조용입니다.
- 실제 발송, 결재, 계약, 업체 선정, 예산 집행, 법률 판단, 민원 처리 결론은 AI나 자동화 시스템이 수행하지 않습니다.
- 확인되지 않은 값은 [확인 필요], null, placeholder로 표시합니다.
- 판단이 애매하면 더 보수적으로 처리합니다.
- 문서 내용 생성과 파일 서식 적용을 분리합니다.

## 프로젝트 핵심 우선순위

- 이 프로젝트의 최우선 자동화 대상은 이메일이 아니라 HWPX 기반 행정 보고서 템플릿입니다.
- 실무에서 가장 많이 사용하는 핵심 문서는 원페이지 보고서, 추진계획서, 결과보고서, 검토보고서입니다.
- 공문과 이메일은 보조 자동화 대상이며, HWPX 보고서 자동화보다 우선하지 않습니다.
- 기존 HWP/HWPX 정형 양식을 복사하여 placeholder 기반 템플릿으로 바꾸고, AI 출력 JSON을 치환하여 초안 문서를 만드는 것이 핵심 방향입니다.
- 실제 기관 양식 원본은 Git에 추가하지 않습니다.
- 실제 HWP/HWPX 원본, 실제 공문 원문, 실제 개인정보는 저장소에 추가하지 않습니다.
- 보고서 자동화에서는 형식, 폰트, 자간, 줄간격, 번호체계, 표 양식, 붙임 표기가 문장 생성만큼 중요합니다.
- AI는 내용 구조와 문장 초안을 생성하고, HWPX 템플릿과 렌더러는 서식 유지와 placeholder 치환을 담당합니다.

| 우선순위 | 대상 | 출력 형식 | 비고 |
|---|---|---|---|
| 1 | 원페이지 보고서 | HWPX | 최우선 자동화 대상 |
| 2 | 추진계획서 | HWPX | 계획 수립 문서 |
| 3 | 결과보고서 | HWPX | 계획 대비 결과 정리 |
| 4 | 검토보고서 | HWPX | 내부 검토자료 |
| 5 | 조사표 | XLSX | 보조 자동화 대상 |
| 6 | 업체 메일 | Email | 보조 초안 작성 |

## 문서 유형별 문체 원칙

### HWPX 보고서 문체

원페이지 보고서, 추진계획서, 결과보고서, 검토보고서는 다음 문체를 우선합니다.

- 개조식 중심
- 짧은 문장
- 명사형 표현
- 불필요한 조사 최소화
- 구어체 지양
- 행정문구 사용
- 확정되지 않은 정보는 [확인 필요]
- 예산, 일정, 담당자, 수량은 임의 생성 금지
- 과장된 기대효과 지양
- 사람이 최종 검토할 수 있는 초안 형태 유지

권장 표현 예시:

- [업무 목적]을 위하여 [대상]에 대한 [요청 내용]을 추진하고자 함
- [활용 목적]을 위한 기초자료로 활용하고자 함
- [제출기한]까지 [제출방법]으로 제출하여 주시기 바랍니다
- 세부 추진 여부는 내부 검토 및 관련 절차에 따라 결정 예정
- 실제 발송 전 담당부서 검토 및 내부 승인 필요

### 공문 문체

공문은 다음 문체를 사용합니다.

- 공문체
- 요청사항, 제출기한, 작성 유의사항 중심
- 번호체계 유지
- 붙임 표기 유지
- 수신처, 시행일자, 문서번호 등 확인되지 않은 값은 [확인 필요]
- 개인정보와 내부 운영정보 제외 안내 포함

### 이메일 문체

이메일은 공문체가 아니라 정중한 실무 메일체를 사용합니다.

- 지나치게 딱딱한 공문체를 피함
- 평이하고 정중한 문장 사용
- "안녕하세요.", "확인 부탁드립니다." 등 실무 메일 표현 허용
- 다만 계약, 업체 선정, 예산 집행이 확정된 것처럼 표현하지 않음
- 개인정보, 내부자료, 민감정보 제외 요청 문구 포함
- 개인 연락처보다 공용 연락처 placeholder 우선 사용

### 협조 요청 메일과 협조 요청 공문 구분

- 협조 요청 공문: 공문체, 번호체계, 붙임, 작성 유의사항 중심
- 협조 요청 메일: 정중한 실무 메일체, 요청 배경과 회신 방법 중심
- 두 문체를 혼용하지 않음

### 결과보고서 문체

결과보고서는 기존 계획과 결과를 대응시켜 정리합니다.

- 추진개요
- 추진결과
- 주요성과
- 문제점 및 개선사항
- 향후계획

주의:

- 참여 인원, 실적 수치, 예산 집행액은 임의 생성하지 않음
- 사용자가 제공하지 않은 결과값은 [확인 필요]로 표시
- 성과를 과장하지 않음

## 기존 HWP/HWPX 보고서 템플릿 활용 원칙

- 실무에서 사용하는 기존 HWP/HWPX 보고서 양식은 원본을 직접 수정하지 않습니다.
- 원본 양식은 Git에 추가하지 않습니다.
- 테스트가 필요한 경우 복사본을 만들고 실제 내용을 제거한 뒤 placeholder를 삽입합니다.
- placeholder 기반 테스트 템플릿은 처음에는 로컬에서만 관리합니다.
- Git에 HWP/HWPX 파일을 추가하려면 별도 검수와 사용자 명시적 승인이 필요합니다.
- 실제 기관명, 문서번호, 결재선, 직인, 로고, 담당자명은 테스트 템플릿에 넣지 않습니다.
- 모든 동적 값은 {{title}}, {{body_section_1}}, {{overview_table}}, {{attachments}} 같은 placeholder로 처리합니다.
- HWPX 보고서 자동화에서는 기존 양식의 폰트, 자간, 줄간격, 표 양식, 문단 배치 유지가 핵심입니다.
- 실제 기관 양식으로 넘어가기 전에는 안전 절차와 체크리스트를 먼저 통과해야 합니다.

## 공문서 작성 요령 PDF 반영 원칙

- 사용자가 공문서 작성 요령 PDF를 제공하면 문체 품질 향상을 위한 참고자료로 활용할 수 있습니다.
- PDF 원문을 그대로 학습시키거나 긴 원문을 저장소에 복사하지 않습니다.
- PDF에서 추출할 수 있는 것은 작성 원칙, 문장 패턴, 번호체계, 지양 표현, 권장 표현, 예시 구조입니다.
- PDF에 민감정보, 내부자료, 비공개 정보가 포함되어 있으면 외부 AI 처리 대상에서 제외합니다.
- PDF 내용을 반영할 때는 보고서 문체, 계획서 문체, 결과보고서 문체, 공문체, 이메일체를 구분합니다.
- PDF에서 가져온 예시는 실제 사례처럼 쓰지 않고 placeholder 기반 패턴으로 변환합니다.
- 공문서 작성 요령은 HWPX 보고서 자동화 품질을 높이기 위한 참고 규칙으로 반영합니다.

## 구현 방향 재확인

- 보고서 HWPX 자동화가 최우선입니다.
- Email 렌더러는 보조 기능입니다.
- XLSX 조사표는 보조 자동화 대상입니다.
- HWPX 보고서 템플릿 자동화가 안정화되기 전에는 실제 이메일 자동화, Make.com 연동, API 자동 호출을 서두르지 않습니다.
- 자동화보다 안전한 초안 생성, 템플릿 치환, 사람 검토 흐름이 우선입니다.

## 현재 진행 기준

- Phase 1 문서화와 placeholder 기반 HWPX 보고서 4종 검증은 완료된 상태로 봅니다.
- 현재 작업 축은 실제 양식 수동 리허설 사용자 확인 패킷과 gap log 빈 양식 정리입니다.
- Phase 2는 운영 자동화가 아니라 비식별 입력 정규화, 보안 필터, HWPX payload 매핑, dry-run, 사람 검토 흐름을 안정화하는 단계입니다.
- `normalizers/` 코드는 placeholder fixture 기반 최소 PoC이며 운영 코드가 아닙니다.
- mapped HWPX output은 로컬 검증 산출물이며 실제 업무용 문서가 아닙니다.
- Phase 2 운영 문서 묶음은 최종 정리 단계까지 도달한 상태입니다.
- normalizers 회귀 테스트 묶음 재검증까지 완료된 상태입니다.
- Phase 2 최소 PoC는 문서 기준 마무리 가능 상태로 판단합니다.
- Phase 3 진입 조건 및 안전 게이트 문서화까지 완료된 상태입니다.
- Phase 3 저장소 밖 HWPX 취급 및 사용자 수동 preview 기준 구체화까지 완료된 상태입니다.
- Phase 3 상태별 중단 기준을 반복 운영 문서에 반영한 상태입니다.
- Phase 3 운영 문서 묶음 통합 점검까지 완료된 상태입니다.
- Phase 3 외부 연동 필요성과 보류 기준 검토까지 완료된 상태입니다.
- Phase 3 로그와 감사 추적 기준 구체화까지 완료된 상태입니다.
- Phase 3 테스트 계정과 테스트 데이터 기준 구체화까지 완료된 상태입니다.
- Phase 3 실제 원문 차단과 비식별 입력 확인 절차 구체화까지 완료된 상태입니다.
- Phase 3 사용자 preview와 사람 승인 지점 통합 기준 구체화까지 완료된 상태입니다.

## 6. 보안등급 기준

문서 후보를 다룰 때는 A/B/C/D 기준을 사용합니다.

등급:
- A: 공개 가능
- B: 비식별 후 사용 가능
- C: 내부 참고만 가능
- D: 외부 처리 금지

처리 원칙:
- A/B등급만 비식별 샘플 후보로 검토합니다.
- C등급은 외부 AI 입력, Custom GPT Knowledge 후보, 비식별 샘플 후보에서 제외합니다.
- D등급은 외부 처리 금지이며 샘플 작성 대상에서도 제외합니다.
- A/B 애매하면 B등급으로 분류합니다.
- B/C 애매하면 C등급으로 분류합니다.
- C/D 애매하면 D등급으로 분류합니다.

자동 D등급 후보:
- 민원인 실명, 연락처, 주소
- 민원번호, 접수번호, 사건번호
- 차량번호, 호출부호
- 관서별 차량 또는 장비 현황
- 순찰구역, 배치 위치, 운영시간
- 감사, 징계, 인사, 수사, 보안 관련 문서
- 대외비, 비공개, 내부전용 표시 문서
- 내부망 주소, 시스템명, 계정, 비밀번호, API 키
- 실제 업체 견적 금액, 계약 조건, 평가 의견
- 특정 개인, 기관, 업체, 사건을 쉽게 추정할 수 있는 문맥

## 9. Custom GPT 관련 규칙

핵심 파일:
- prompts/custom_gpt_instructions_v0.1.md
- prompts/conversation_starters.md

금지:
- 기존 공문 원문을 Knowledge에 바로 업로드 안내 금지
- 민감자료를 Knowledge 후보로 추천 금지
- 내부자료, 대외비, 민원자료, 관서별 현황자료를 Knowledge에 넣지 않음
- Custom GPT가 실제 발송, 결재, 계약, 업체 선정, 예산 집행을 하도록 설계하지 않음

원칙:
- Custom GPT는 초안 작성 보조
- 모든 외부 발송문은 초안
- 부족한 정보는 [확인 필요]
- 위험 요청은 제한하고 안전한 대체안 제시
- 민원인 개인정보를 업체에 전달하는 요청은 그대로 수행하지 않음
