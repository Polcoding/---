# Codex 작업 규칙

## 우선 원칙

기능 구현보다 안전한 문서화와 테스트 기준 정리가 우선입니다.

이 저장소에서 Codex는 공공기관 행정문서 AI 자동화 시스템의 1단계 준비물을 작성합니다. 실제 자동화 구현, API 연동, 이메일 발송, 기존 공문 원문 학습은 이번 단계의 범위가 아닙니다.

## 작성 언어

- 모든 문서는 한국어로 작성합니다.
- 개발자가 이해하기 쉬운 실무 문서체를 사용합니다.
- 지나치게 마케팅스럽거나 법률문서처럼 어려운 표현은 피합니다.
- 보안 원칙은 단호하게 작성합니다.

## 예시 작성 규칙

- 실제 사람 이름을 만들지 않습니다.
- 실제 전화번호, 주소, 이메일, 주민등록번호, 계좌번호를 만들지 않습니다.
- 실제 기관명, 실제 관서명, 실제 차량번호, 실제 장비 수량을 만들지 않습니다.
- 실제 업체 견적, 실제 내부 운영현황, 실제 민원 세부내용을 만들지 않습니다.
- 예시는 `[기관 A]`, `[업체 A]`, `[대상자 A]`, `[확인 필요]` 같은 placeholder만 사용합니다.

## 보안상 금지되는 작업

- 보안정책을 회피하는 문구를 작성하지 않습니다.
- 내부자료를 외부 AI로 처리하도록 안내하지 않습니다.
- 기존 공문 원문을 Custom GPT Knowledge에 업로드하라고 안내하지 않습니다.
- Make.com, Gmail, Outlook, OpenAI API 실제 연동 코드를 작성하지 않습니다.
- API 키, 인증 토큰, 환경변수 예시 값을 만들지 않습니다.
- 실제 발송, 결재, 계약, 업체 선정, 예산 집행, 법률 판단, 민원 처리 결론을 자동화하지 않습니다.

## 문서 구조 유지

지정된 디렉터리 구조를 유지합니다.

```text
docs/
prompts/
examples/
checklists/
```

새 문서를 추가할 때는 기존 목적과 충돌하지 않게 작성하고, README의 문서 목록도 함께 갱신합니다.

## 변경 후 자체 점검 기준

변경 후 다음을 확인합니다.

- 모든 문서가 한국어인지
- 실제 개인정보처럼 보이는 값이 없는지
- 실제 기관 운영정보처럼 보이는 값이 없는지
- 실제 관서명, 차량번호, 장비현황처럼 보이는 값이 없는지
- Custom GPT Instructions가 그대로 복사해 사용할 수 있는지
- 테스트 케이스가 5개 이상인지
- 위험 요청 예시가 포함되어 있는지
- 평가표가 0~2점 기준인지
- API 연동이나 Make.com 구현 코드가 없는지
- 기존 공문 원문 업로드를 권장하지 않는지
- 변경 이유와 생성ㆍ수정 파일 목록을 완료 보고에 포함했는지

## 완료 기준

- 지정된 Markdown 파일이 모두 존재합니다.
- README만 봐도 프로젝트 의도와 1단계 범위를 이해할 수 있습니다.
- 보안 정책과 비식별화 규칙이 명확합니다.
- Custom GPT 테스트에 필요한 Instructions, Conversation Starters, 테스트 케이스, 평가표가 준비되어 있습니다.
- 모든 예시는 가상 정보 또는 placeholder만 사용합니다.
- 변경 후 보안 검수를 수행하고, 실제 정보처럼 보이는 값이 있으면 placeholder로 바꿉니다.
