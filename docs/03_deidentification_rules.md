# 비식별화 규칙

## 목적

개인정보와 민감정보를 외부 AI에 그대로 전달하지 않도록 치환 기준을 정의합니다.

## 비식별화 대상

- 실명
- 휴대전화번호
- 개인 이메일
- 주소
- 주민등록번호
- 계좌번호
- 차량번호
- 관서명
- 업체명
- 문서번호
- 사건번호
- 민원번호
- 구체 예산액
- 구체 날짜
- 내부 시스템명
- 계정, 비밀번호, API 키

## 치환 규칙

| 원래 정보 유형 | 치환 표현 |
|---|---|
| 실명 | `[대상자 A]`, `[담당자 A]` |
| 휴대전화번호 | `[연락처 비공개]` |
| 개인 이메일 | `[이메일 비공개]` |
| 주소 | `[주소 비공개]` |
| 차량번호 | `[차량 식별정보 비공개]` |
| 관서명 | `[기관 A]`, `[관서 A]` |
| 업체명 | `[업체 A]` |
| 문서번호 | `[문서번호 확인 필요]` |
| 사건번호 | `[사건번호 비공개]` |
| 민원번호 | `[민원번호 비공개]` |
| 구체 예산액 | `[예산 확인 필요]` |
| 구체 날짜 | `[일자 확인 필요]` |
| 내부 시스템명 | `[내부 시스템명 비공개]` |

## 나쁜 예시

```text
[이름삭제] 민원인이 [구체 날짜]에 [특정 관서]에 방문하여 [특정 장비] 부족을 신고함.
```

문제점:

- 구체 날짜가 placeholder로 바뀌었더라도 사건의 흐름이 여전히 구체적입니다.
- 사건의 맥락이 지나치게 구체적입니다.
- 특정 장소와 특정 장비를 연상할 수 있습니다.

## 좋은 예시

```text
[대상자 A]가 [특정 시점]에 [대상 기관] 관련 사항을 제기함에 따라 해당 현황을 점검하고자 함.
```

## 더 좋은 학습용 문체 샘플

```text
[업무 발생 배경]에 따라 [대상 기관]의 [점검 대상] 현황을 확인하고, 향후 관리 개선을 위한 기초자료로 활용하고자 함.
```

## 단순 마스킹과 문맥 비식별화의 차이

단순 마스킹은 일부 문자만 가리는 방식입니다. 하지만 문맥이 구체적이면 누가, 언제, 어디서, 어떤 사건을 말하는지 추정될 수 있습니다.

문맥 비식별화는 사건의 고유성을 줄이고 업무 구조만 남기는 방식입니다. 이 프로젝트에서는 문맥 비식별화를 우선합니다.

## 학습용 샘플 작성 원칙

- 실제 사건성이 사라져야 합니다.
- 업무 문체와 구조만 남겨야 합니다.
- 실제 기관, 사람, 차량, 장비, 예산, 날짜가 떠오르지 않아야 합니다.
