# Phase 2 보안 필터 요구사항

## 목적

이 문서는 입력 정규화 결과를 AI 초안 생성 또는 렌더러로 넘기기 전에 차단, 보류, 검토, 진행 여부를 판단하기 위한 보안 필터 요구사항입니다.

보안 필터는 실제 원문을 저장하거나 실제 개인정보를 처리하기 위한 기능이 아닙니다. 비식별 입력만 초안 생성 흐름으로 보내고, 위험 신호가 있는 입력은 사람 검토 또는 차단으로 라우팅하기 위한 게이트입니다.

## 적용 위치

```text
사용자 입력
→ 입력 정규화
→ 보안 필터
→ 업무 유형 분류 보정
→ AI 초안 생성
→ HWPX/XLSX/Markdown/Email 렌더링
→ 사람 검토
```

보안 필터는 최소 두 번 적용합니다.

| 시점 | 목적 |
|---|---|
| 초안 생성 전 | 실제 원문, 개인정보, 내부정보가 AI 입력으로 넘어가지 않도록 차단 |
| 렌더링 전 | AI 출력 JSON에 임의 생성값, 확정 표현, 위험 문구가 섞였는지 재검토 |

## 입력

보안 필터의 기본 입력은 `docs/61_input_normalization_schema.md`의 정규화 구조입니다.

필수 확인 필드:

- `user_intent_summary`
- `candidate_document_type`
- `content_inputs`
- `known_values`
- `missing_fields`
- `security_flags`
- `routing_decision`
- `human_review_required`

원문 전체는 보안 필터 입력으로 저장하지 않습니다.

## 출력

보안 필터는 다음 구조를 반환합니다.

```json
{
  "security_grade": "B",
  "risk_level": "medium",
  "routing_status": "needs_security_review",
  "detected_flags": [],
  "blocked_reasons": [],
  "review_notes": [],
  "allowed_next_action": "human_review_before_draft"
}
```

## A/B/C/D 등급과 risk_level 연결

| 보안등급 | risk_level | 기본 라우팅 | 설명 |
|---|---|---|---|
| A | low | `ready_for_draft` | 공개 가능 수준의 일반 구조 또는 placeholder 입력 |
| B | medium | `needs_security_review` 또는 `ready_for_draft` | 비식별 후 제한적으로 초안 생성 가능 |
| C | high | `needs_security_review` | 내부 참고만 가능, 외부 AI 입력 금지 원칙 |
| D | blocked | `blocked` | 외부 처리 금지, 초안 생성ㆍ렌더링 중단 |

판단이 애매하면 더 보수적인 등급을 적용합니다.

- A/B가 애매하면 B
- B/C가 애매하면 C
- C/D가 애매하면 D

## routing_status 기준

| 상태 | 사용 조건 | 다음 동작 |
|---|---|---|
| `ready_for_draft` | 실제 식별정보 없음, placeholder 기반, 필수값 또는 누락값 분리 완료 | 초안 생성 가능 |
| `needs_more_input` | 보안 문제는 낮지만 필수값 부족 | 사용자에게 확인 질문 |
| `needs_security_review` | 실제값 의심, 내부정보 가능성, B/C 경계 | 사람 검토 후 진행 여부 결정 |
| `blocked` | 개인정보, 실제 원문, D등급 정보, 자동화 금지 업무 포함 | 처리 중단 및 비식별 재요청 안내 |

## 탐지해야 할 위험 신호

### 개인정보와 고유식별정보

- 실명과 직위가 결합된 문맥
- 전화번호, 휴대전화번호 형식
- 이메일 주소 형식
- 주소 또는 상세 위치 형식
- 주민등록번호, 여권번호, 외국인등록번호 등 고유식별정보 형식
- 계좌번호, 카드번호 등 금융 식별정보 형식

### 문서ㆍ사건 식별정보

- 실제 문서번호 의심 형식
- 접수번호, 민원번호, 사건번호 의심 형식
- 고유한 내부 관리번호 의심 형식
- 실제 공문 제목과 발신ㆍ수신 정보가 함께 있는 문맥

### 내부 운영정보

- 내부망 URL 또는 내부 시스템명
- 계정, 비밀번호, API 키, 토큰
- 보안 설정, 접근 권한, 내부 절차 세부값
- 관서별 실제 운영 현황, 차량번호, 장비 세부 현황

### 예산ㆍ계약ㆍ업체 정보

- 실제 금액, 견적, 계약 조건
- 업체 평가 의견, 선정 사유, 순위
- 계약 확정, 예산 확보, 업체 선정 단정 표현
- 특정 업체를 유리하거나 불리하게 평가하는 문맥

### 민감 업무

- 감사, 징계, 수사, 조사, 분쟁
- 민원 처리 세부 내용
- 인사, 복무, 신상, 건강, 장애 관련 내용
- 대외비, 비공개, 내부전용 표시

## 안전한 설명 문맥과 실제값 의심 문맥 구분

| 구분 | 안전한 문맥 | 실제값 의심 문맥 |
|---|---|---|
| placeholder | `[기관명]`, `[담당자]`, `[확인 필요]` | 실제 이름, 실제 기관명, 실제 연락처 |
| 누락값 | `예산 확인 필요`, `담당자 확인 필요` | 특정 금액이나 담당자명이 제시됨 |
| 정책 설명 | "개인정보 포함 시 차단" | 특정 개인의 정보 처리 요청 |
| 문서 구조 | "붙임 1. 자료 목록" | 실제 붙임 파일명, 실제 문서번호 포함 |
| 계약 표현 | "계약 여부는 사람 검토" | "업체 선정 완료", "계약 확정" |

안전한 문맥이라도 실제값처럼 보이는 숫자ㆍ고유명사ㆍ연락처 형식이 있으면 `needs_security_review` 이상으로 올립니다.

## 차단 기준

다음 중 하나라도 명확하면 `blocked`로 처리합니다.

- 실제 개인정보 또는 고유식별정보 포함 의심
- 실제 민원, 사건, 접수, 문서번호 포함 의심
- 실제 원문 전체 또는 실제 공문 본문 붙여넣기 의심
- C/D등급 문서를 외부 AI에 입력하려는 요청
- 실제 계약, 업체 선정, 예산 집행, 법률 판단을 확정하려는 요청
- 실제 이메일 발송, 결재, 계약, 예산 집행 자동화 요청
- API 키, 계정, 비밀번호, 내부망 URL 포함 의심

## 보류 기준

다음은 즉시 차단보다 사람 검토가 먼저 필요한 경우입니다.

- A/B등급 판단이 애매한 입력
- 문서 구조는 안전하지만 실제값으로 보이는 숫자가 섞인 입력
- 내부 운영정보인지 공개 정보인지 불명확한 입력
- 예산, 일정, 실적, 담당자 값이 실제값인지 placeholder인지 불명확한 입력
- 민감 업무 키워드가 있으나 실제 세부 내용은 없는 입력

## 진행 가능 기준

다음 조건을 모두 만족하면 `ready_for_draft`로 둘 수 있습니다.

- 실제 개인정보 없음
- 실제 원문 없음
- 실제 기관명, 담당자명, 문서번호 없음
- 예산, 일정, 실적, 수량은 placeholder 또는 `missing_fields`로 분리
- 문서 유형이 허용 범위 안에 있음
- 금지 업무 자동화 요청 없음
- `human_review_required`가 `true`

## 렌더링 전 재검토

AI 출력 JSON 또는 렌더러 입력에는 다음 검사를 다시 적용합니다.

- `[확인 필요]` 또는 `null`이어야 할 값이 임의 생성되지 않았는가
- 예산, 일정, 실적, 담당자, 수량이 사용자가 제공하지 않았는데 채워지지 않았는가
- 실제 연락처, 이메일, 문서번호 형식이 새로 생기지 않았는가
- 계약, 업체 선정, 예산 확보, 법률 적합성을 단정하지 않았는가
- HWPX output과 로컬 템플릿이 Git 제외 상태인가

## 사용자 안내 문구 원칙

차단 또는 보류 시에는 실제 입력 내용을 반복하지 않습니다.

권장 문구:

- "입력에 실제 개인정보 또는 내부정보가 포함되었을 가능성이 있어 처리하지 않습니다."
- "실제값을 제거하고 `[확인 필요]` 또는 placeholder로 바꾼 뒤 다시 요청해 주세요."
- "이 요청은 사람이 보안 검토한 뒤 진행 여부를 판단해야 합니다."

지양 문구:

- 감지된 실제값을 그대로 다시 보여주는 문구
- 특정 개인정보 유형을 과도하게 상세히 재현하는 문구
- 차단 우회 방법을 알려주는 문구

## 구현 전 완료 조건

- A/B/C/D 등급과 `risk_level` 연결 기준 확정
- `routing_status` 판정 기준 확정
- 차단 패턴과 보류 기준 확정
- 안전한 placeholder 문맥과 실제값 의심 문맥 구분 기준 확정
- 렌더링 전 재검토 기준 확정
- 실제 원문, 개인정보, 기관정보 미사용 확인

## 다음 단계

이 문서를 기준으로 입력 정규화 테스트 케이스를 작성합니다.

테스트 케이스는 실제 개인정보나 실제 기관명을 쓰지 않고, 안전한 placeholder 입력, 누락값 입력, 보류 입력, 차단 입력을 각각 분리해 작성합니다.
