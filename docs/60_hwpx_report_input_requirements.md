# HWPX 보고서 4종 입력 요구사항

## 목적

이 문서는 입력 정규화 스키마를 만들기 전에, HWPX 보고서 4종이 최소로 필요로 하는 입력값과 누락값 처리 정책을 정리합니다.

대상 문서 유형은 다음 4종입니다.

- `one_page_report`
- `project_plan`
- `result_report`
- `review_report`

## 공통 입력 원칙

- 실제 원문을 그대로 입력하지 않습니다.
- 실제 개인정보, 내부 운영정보, 민원정보, 차량번호, 실제 문서번호를 입력하지 않습니다.
- 입력값이 없으면 임의 생성하지 않습니다.
- 예산, 일정, 실적, 담당자, 수량은 사용자가 제공하지 않으면 `[확인 필요]`로 둡니다.
- AI는 초안 작성만 수행하며 실제 발송, 결재, 계약, 업체 선정, 예산 집행, 법률 판단은 수행하지 않습니다.
- `security_review`, `missing_fields`, `draft_status`, `human_review_required`는 모든 문서 유형에서 유지합니다.

## 공통 필수 필드

| 필드 | 용도 | 누락 시 처리 |
|---|---|---|
| `document_type` | 문서 유형 분기 | 렌더링 중단 또는 `[확인 필요]` |
| `output_targets` | 출력 형식 지정 | `hwpx` 없으면 HWPX 렌더링 제외 |
| `security_review` | 보안 검토 결과 | 없으면 렌더링 중단 |
| `missing_fields` | 확인 필요 항목 | 없으면 빈 배열 허용 |
| `draft_status` | 초안 상태 | `draft` 권장 |
| `human_review_required` | 사람 검토 필요 여부 | `true` 고정 권장 |
| `documents[0].title` | 문서 제목 | `[제목 확인 필요]` |
| `renderer_hints` | 템플릿/placeholder 힌트 | 없으면 렌더러 기본값 사용 가능 |

## 원페이지 보고서

### 최소 입력 필드

| 필드 | 용도 | 누락 시 처리 |
|---|---|---|
| `title` | 보고 제목 | `[보고 제목 확인 필요]` |
| `report_summary` | 보고 개요 | `[보고일자ㆍ담당부서ㆍ검토자ㆍ보고목적 확인 필요]` |
| `background` | 추진 배경 | `[업무 발생 배경 및 관련 근거 확인 필요]` |
| `main_points` | 주요 내용 | `[주요 내용 확인 필요]` |
| `review_opinion` | 검토 의견 | `[검토 의견 확인 필요]` |
| `issues_or_considerations` | 문제점 및 고려사항 | `[고려사항 확인 필요]` |
| `next_steps` | 향후 계획 | `[향후 조치 확인 필요]` |
| `action_items` | 조치 필요사항 | `[조치 필요사항 확인 필요]` |

### 렌더링 주의

- `report_summary`는 객체를 여러 줄로 펼치지 않고 짧은 확인 문구로 축약합니다.
- 원페이지 보고서는 공간이 좁으므로 긴 배열은 핵심 항목만 남깁니다.
- 실제 세부값이 많으면 별도 보고서나 붙임으로 분리합니다.

## 추진계획서

### 최소 입력 필드

| 필드 | 용도 | 누락 시 처리 |
|---|---|---|
| `title` | 계획서 제목 | `[사업명] 추진계획` 또는 `[사업명 확인 필요] 추진계획` |
| `background` | 추진 배경 | `[업무 발생 배경 확인 필요]` |
| `purpose` | 추진 목적 | `[활용 목적 확인 필요]` |
| `overview` | 추진 개요 | `[사업명ㆍ추진기간ㆍ추진대상ㆍ추진방식ㆍ소요예산 확인 필요]` |
| `main_contents` | 주요 내용 | `[주요 내용 확인 필요]` |
| `detailed_plan` | 세부 추진계획 | `[단계별 세부 추진내용 확인 필요]` |
| `schedule` | 추진 일정 | `[추진 단계ㆍ기간ㆍ세부 일정 확인 필요]` |
| `budget` | 소요 예산 | `[소요예산ㆍ예산 항목ㆍ산출 기준 확인 필요]` |
| `expected_effects` | 기대 효과 | `[기대효과 확인 필요]` |
| `review_items` | 검토 사항 | `[검토사항 확인 필요]` |
| `future_plan` | 향후 계획 | `[향후계획 확인 필요]` |

### 렌더링 주의

- `overview`, `schedule`, `budget`, `detailed_plan`은 긴 객체/배열이면 한 줄 확인 문구로 축약합니다.
- 4번, 8번, 9번 내용란은 다른 항목과 맞춰 앞쪽 `-` 없이 표시합니다.
- 예산과 일정은 사용자가 제공하지 않으면 절대 임의 생성하지 않습니다.

## 결과보고서

### 최소 입력 필드

| 필드 | 용도 | 누락 시 처리 |
|---|---|---|
| `title` | 결과보고서 제목 | `[사업명 확인 필요] 결과보고` |
| `linked_plan_reference` | 기존 계획 연결 | `[기존 추진계획서 참조 확인 필요]` |
| `overview` | 추진 개요 | `[사업명ㆍ추진기간ㆍ추진대상ㆍ담당부서 확인 필요]` |
| `planned_items` | 계획 항목 | `[계획 항목 확인 필요]` |
| `actual_results` | 추진 결과 | `[추진결과ㆍ실적 수치ㆍ참여 인원ㆍ예산 집행액 확인 필요]` |
| `comparison_to_plan` | 계획 대비 결과 | `[계획 대비 결과 확인 필요]` |
| `main_outcomes` | 주요 성과 | `[주요성과 확인 필요]` |
| `issues` | 문제점 | `[문제점 확인 필요]` |
| `improvements` | 개선사항 | `[개선사항 확인 필요]` |
| `future_plan` | 향후 계획 | `[향후계획 확인 필요]` |

### 렌더링 주의

- 기존 추진계획서 원문을 그대로 입력하지 않습니다.
- 실적 수치, 참여 인원, 예산 집행액은 사용자가 제공하지 않으면 임의 생성하지 않습니다.
- 계획 대비 결과는 실제 계획 항목과 연결되는 값이 없으면 `[확인 필요]`로 둡니다.

## 검토보고서

### 최소 입력 필드

| 필드 | 용도 | 누락 시 처리 |
|---|---|---|
| `title` | 검토보고서 제목 | `[검토 대상 확인 필요] 검토보고` |
| `review_background` | 검토 배경 | `[검토배경 확인 필요]` |
| `review_scope` | 검토 범위 | `[검토범위 확인 필요]` |
| `review_items` | 검토 항목 | `[검토항목ㆍ관련 기준ㆍ검토 상태 확인 필요]` |
| `review_opinion` | 검토 의견 | `[검토의견 확인 필요]` |
| `risks` | 위험요소 | `[위험요소 확인 필요]` |
| `required_reviews` | 필요 검토 | `[법무ㆍ계약ㆍ개인정보 검토 필요 여부 확인 필요]` |
| `next_actions` | 후속조치 | `[후속조치 확인 필요]` |

### 렌더링 주의

- 법률적으로 문제없다는 확정 표현을 사용하지 않습니다.
- 계약, 예산, 개인정보 관련 판단은 필요 검토 항목으로 남깁니다.
- `risk_level`이 `high` 또는 `blocked`이면 렌더링하지 않습니다.

## 누락값 처리 정책

| 상황 | 처리 |
|---|---|
| 제목 누락 | 문서 유형별 제목 placeholder 사용 |
| 본문 핵심값 누락 | `[확인 필요]` 포함 문구 사용 |
| 배열 누락 | `[확인 필요]` 또는 빈 문자열 중 템플릿별 선택 |
| 메타 검토값 누락 | `missing_fields`에 기록 |
| 보안 검토값 누락 | 렌더링 중단 권장 |
| 예산, 일정, 실적, 담당자, 수량 누락 | 임의 생성 금지, `[확인 필요]` |
| 실제 원문 포함 | 외부 처리 중단 |
| C/D등급 문서 | 렌더링 및 외부 AI 처리 중단 |

## 입력 정규화로 넘길 항목

Phase 2 입력 정규화 설계에서는 다음 항목을 구조화해야 합니다.

- 사용자가 원하는 문서 유형
- 보고 또는 계획 목적
- 주요 내용
- 일정 또는 기한
- 예산 또는 비용 관련 정보
- 담당부서 또는 검토 주체
- 결과 또는 실적 수치
- 검토가 필요한 위험요소
- 첨부 또는 붙임 필요 여부
- 누락된 정보 목록
- 보안등급 판단 결과

## 결론

입력 정규화 스키마를 만들기 전에, HWPX 보고서 4종은 위 필드를 기준으로 최소 입력값과 누락값 정책을 확정해야 합니다.

현재 단계에서는 실제 입력 원문을 사용하지 않고 placeholder 기반 샘플과 정책 문서로만 정리합니다.
