# 사용자 운영 A-to-Z 안내

## 목적

이 문서는 사용자가 HWPX 보고서 자동화 PoC를 다룰 때 직접 해야 하는 일과 Codex가 처리할 수 있는 일을 한 번에 보기 위한 안내입니다.

실제 업무 원문, 실제 개인정보, 실제 기관명, 실제 문서번호, 실제 HWPX/HWP 양식 원본, 실제 표 데이터는 이 문서에 기록하지 않습니다.

## 현재 기준

현재 저장소는 실제 업무 운영 시스템이 아니라 비식별 placeholder 기반 로컬 PoC입니다.

```text
비식별 요청
-> 입력 정규화
-> 보안 필터
-> HWPX payload mapper
-> payload validation
-> HWPX renderer dry-run
-> 사용자 검토
```

실제 발송, 결재, 계약, 업체 선정, 예산 집행, 법률 판단은 사람이 수행합니다.

## 역할 구분

| 표시 | 의미 | 처리 기준 |
|---|---|---|
| `[사용자 확인 필요]` | 사람이 직접 판단하거나 확인해야 하는 값 | Codex가 임의 작성하지 않음 |
| `[Codex 처리 가능]` | 비식별 placeholder 입력을 바탕으로 구조화 가능한 항목 | 초안 구조화, 누락값 목록화, 보안 주의문구 정리 가능 |
| `[보류]` | 현재 단계에서 자동화하지 않는 항목 | 실제 HWPX 원본, 실제 표 데이터, 외부 연동은 실행하지 않음 |

## A. 시작 전 확인

`[사용자 확인 필요]`

- 만들 문서 유형 확인: `one_page_report`, `project_plan`, `result_report`, `review_report`
- 실제 원문, 개인정보, 기관명, 문서번호, 실제 예산액이 제거되었는지 확인
- 모르는 값은 `[확인 필요]`로 둘지 내부 검토 후 결정
- 실제 HWPX 원본을 저장소에 넣지 않았는지 확인

`[Codex 처리 가능]`

- 비식별 입력 구조 확인
- 문서 유형별 필요한 필드 안내
- 누락 가능 항목을 `missing_fields` 검토 목록으로 분리

`[보류]`

- 실제 원문 기반 자동 작성
- 실제 기관 양식 원본 Git 추가
- 실제 개인정보나 문서번호가 포함된 테스트

## B. 입력 작성

사용자는 `docs/84_hwpx_report_user_input_templates.md`의 문서 유형별 입력 템플릿을 사용합니다.

`[사용자 확인 필요]`

- 보고 목적, 추진 일정, 예산, 실적, 검토 범위처럼 사람이 판단해야 하는 항목
- 표 데이터가 필요한지 여부
- 표 내부 값은 Excel/한셀 분리 후보로 둘지 여부

`[Codex 처리 가능]`

- 개조식 초안 구조화
- 보고서 4종 문체 구분
- 확인 필요 항목 목록화
- 보안 주의문구 유지

`[보류]`

- 실제 일정표, 예산표, 실적표 자동 작성
- 실제 수량, 금액, 대상 목록 입력
- 실제 검토의견 원문 처리

## C. Codex 처리 요청

사용자가 Codex에 요청할 때는 실제값 대신 placeholder를 사용합니다.

권장 요청 형식:

```text
[비식별 업무] [문서 유형] 초안 구조를 정리해 주세요.
실제 원문, 개인정보, 기관명, 문서번호, 예산액은 포함하지 않았습니다.
확인되지 않은 값은 [확인 필요]로 유지해 주세요.
```

`[사용자 확인 필요]`

- 입력에 실제값이 섞이지 않았는지 최종 확인
- 결과물을 실제 결재나 발송에 사용할지 여부는 내부 절차로 판단

`[Codex 처리 가능]`

- 입력 정규화
- 보안 필터링
- HWPX payload 후보 생성
- payload validation
- renderer dry-run 상태 확인

`[보류]`

- 실제 이메일 발송
- Make.com, Gmail, Outlook, OpenAI API 실제 연동
- 결재, 계약, 예산 집행 자동화

## D. 결과 확인

Codex 결과는 초안과 검토 목록입니다. 확정 문서가 아닙니다.

`[사용자 확인 필요]`

- `missing_fields`가 본문 실제값처럼 보이지 않는지 확인
- `[확인 필요]` 항목을 내부 검토 후 유지, 수정, 제외할지 판단
- 보안 검토가 필요한 문서는 렌더링 또는 외부 공유 전 멈춤

`[Codex 처리 가능]`

- `missing_fields` 항목과 관련 위치 정리
- blocked, needs_security_review, template_required 같은 안전 중단 상태 설명
- output과 local template이 Git에 올라가지 않는지 확인 명령 실행

`[보류]`

- `missing_fields`를 실제값으로 자동 확정
- 보안 검토 없는 `review_report` HWPX 렌더링
- 실제 문서번호, 승인자명, 담당자명 입력

## E. 한컴에서 확인할 때

실제 HWPX 파일을 사용자가 열어봐야 하는 단계에서는 다음 항목만 확인합니다.

`[사용자 확인 필요]`

- 파일이 열리는지
- 글자 겹침이 없는지
- 항목 순서가 맞는지
- 줄바꿈과 문단 배치가 자연스러운지
- 표 폭, 여백, 행 높이가 어색하지 않은지
- 남은 `{{placeholder}}`가 없는지
- 내용 앞 bullet 또는 번호 표기가 일관적인지

현재 단계에서는 표 내부 실제 데이터까지 자동화하지 않습니다.

## F. GitHub Desktop 확인

`[사용자 확인 필요]`

- GitHub Desktop Changes에 실제 HWPX 원본이 보이지 않는지 확인
- 로컬 HWPX 템플릿이 Changes에 보이지 않는지 확인
- output HWPX가 Changes에 보이지 않는지 확인

`[Codex 처리 가능]`

- `.gitignore` 규칙 확인
- `git check-ignore`로 local template과 output 제외 여부 확인
- 변경 파일 목록 요약

`[보류]`

- 실제 HWPX 원본을 Git에 추가
- output 파일을 결과물처럼 commit
- 실제 업무 파일명을 저장소 문서에 기록

## G. 이상이 있으면

아래 경우에는 진행을 멈추고 원인을 기록합니다.

- 실제 개인정보 또는 기관정보가 발견됨
- 실제 문서번호, 민원번호, 사건번호 의심값이 발견됨
- HWPX output에 글자 겹침, 항목 누락, placeholder 잔여가 있음
- `[사용자 확인 필요]` 항목을 Codex가 임의 작성함
- `[Codex 처리 가능]` 항목이 실제값 생성 권한처럼 처리됨
- `[보류]` 항목이 자동화 실행 대상으로 처리됨
- GitHub Desktop Changes에 실제 HWPX 원본 또는 output 파일이 표시됨

## H. 지금 사용자가 직접 하지 않아도 되는 일

현재 단계에서 사용자가 직접 하지 않아도 되는 일:

- 실제 기관 양식 원본을 저장소에 넣기
- 실제 개인정보가 있는 원문을 Codex에 붙여넣기
- 실제 표 데이터를 HWPX 안에 자동으로 채우기
- Email/API/Make.com 연동 준비하기
- 실제 발송이나 결재 자동화를 승인하기

## I. 현재 가장 쉬운 진행 순서

1. `CURRENT_STATUS.md`로 현재 작동 범위를 봅니다.
2. `docs/84_hwpx_report_user_input_templates.md`에서 문서 유형별 입력 템플릿을 봅니다.
3. 이 문서의 `[사용자 확인 필요]` 항목만 먼저 확인합니다.
4. 실제값은 넣지 않고 `[확인 필요]` 또는 placeholder로 둡니다.
5. Codex는 구조화, 누락값 목록, 보안 주의문구를 정리합니다.
6. HWPX 파일을 열어봐야 하는 지점이면 사용자가 한컴에서 글자 겹침과 배치를 확인합니다.
7. GitHub Desktop Changes에 HWPX 원본이나 output이 보이면 진행을 멈춥니다.

## J. 완료 기준

- 사용자가 직접 해야 하는 일이 `[사용자 확인 필요]`로 보임
- Codex가 처리할 수 있는 일이 `[Codex 처리 가능]`으로 보임
- 현재 단계에서 하지 않는 일이 `[보류]`로 보임
- 실제 개인정보, 실제 기관정보, 실제 문서번호, 실제 원문을 요구하지 않음
- 실제 HWPX 원본, output, local template을 Git에 추가하지 않음
- 실제 발송, 결재, 계약, 예산 집행, 법률 판단을 자동화하지 않음
