# 채널별 문체 정책

## 목적

이 문서는 HWPX 보고서, 공문, Email 초안의 문체를 구분하여 적용하기 위한 기준을 정리합니다.

모든 예시는 placeholder 기반으로 관리하며, 실제 원문이나 실제 개인정보를 사용하지 않습니다.

## 기본 원칙

- HWPX 보고서는 보고서체와 개조식을 우선합니다.
- Email은 공문체가 아니라 실무 메일체를 우선합니다.
- 협조 요청 공문과 협조 요청 메일은 별도 문체로 작성합니다.
- 보고서 자동화가 본 프로젝트의 최우선 HWPX 자동화 대상입니다.
- 공문, 보고서, 이메일은 목적과 수신 맥락이 다르므로 같은 문장을 그대로 재사용하지 않습니다.
- 실제 발송 또는 제출 전에는 사람이 최종 검토합니다.

## 문서 유형별 문체 요약

| 문서 유형 | 적용 문체 | 핵심 기준 |
|---|---|---|
| HWPX 보고서 | 개조식, 보고서체 | 원페이지 보고서, 추진계획서, 결과보고서, 검토보고서 중심 |
| 공문 | 공문체 | 수신처, 번호체계, 붙임, 제출기한, 작성 유의사항 중심 |
| Email | 정중한 실무 메일체 | 인사말, 요청 배경, 회신 방법, 유의사항, 공용 연락처 중심 |

Email은 HWPX 보고서 자동화보다 우선하지 않는 보조 초안 기능입니다.

## HWPX 보고서 문체

HWPX 보고서는 원페이지 보고서, 추진계획서, 결과보고서, 검토보고서를 중심으로 작성합니다.

- 개조식 중심
- 짧은 문장
- 명사형 표현
- 불필요한 조사 최소화
- 확정되지 않은 내용은 [확인 필요]
- 예산, 일정, 수량, 담당자는 임의 생성 금지
- 검토 의견과 향후 계획은 단정하지 않고 확인 필요 사항을 분리

## 공문 문체

공문은 공식 요청, 제출 요청, 협조 요청, 안내 목적에 맞게 작성합니다.

- 수신자에게 요청사항이 명확히 보이도록 작성
- 제출기한, 제출방법, 작성 유의사항을 분리
- 번호체계는 1. / 가. / 1) / 가) 구조 유지
- 개인정보와 민감정보 제외 안내 포함
- 실제 발송 전 담당부서 검토 및 내부 승인 필요

## Email 문체

Email은 업체 협조 요청, 사용자 회신, 사전 문의 등 실무 연락에 맞게 작성합니다.

- 정중한 실무 메일체 사용
- 공문식 표현을 과도하게 사용하지 않음
- 요청 목적과 요청사항을 짧게 분리
- 계약, 선정, 발주, 예산 집행이 확정된 것처럼 표현하지 않음
- 개인정보ㆍ민감정보 제외 요청 문구 포함
- 공용 연락처 placeholder 사용

## 협조 요청 문체 구분

협조 요청 공문:

- 공식 문서 형식
- 수신처, 제목, 본문, 붙임, 끝. 표시 기준 적용
- 제출기한과 작성 유의사항 중심

협조 요청 메일:

- 정중한 실무 연락 형식
- 제목, 인사말, 요청 목적, 요청사항, 회신기한, 유의사항, 서명 중심
- 실제 자동 발송이 아니라 사람이 검토하는 초안으로 관리

## 금지 및 주의

- 실제 기관명, 업체명, 담당자명, 연락처를 작성하지 않습니다.
- 실제 문서번호, 민원번호, 접수번호, 차량번호를 작성하지 않습니다.
- 실제 예산액, 견적 금액, 계약 조건을 작성하지 않습니다.
- 실제 원문을 복사하거나 요약하여 저장소에 기록하지 않습니다.
- 확인되지 않은 값은 [확인 필요]로 표시합니다.
