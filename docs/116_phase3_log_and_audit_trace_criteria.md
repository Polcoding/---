# Phase 3 로그와 감사 추적 기준

## 목적

Phase 3에서 반복 운영 로그와 향후 외부 연동 로그가 어떤 항목을 공유해야 하는지 정리합니다.

이 문서는 구현 지시서가 아닙니다. 실제 OpenAI API 호출, Make.com 시나리오, GmailㆍOutlookㆍSMTP 연동, 실제 이메일 자동 발송, 로그 수집 코드는 계속 작성하지 않습니다.

## 검토 결론

로그와 감사 추적 기준은 문서 기준으로 구체화할 필요가 있습니다.

다만 현재 단계에서는 다음 변경이 필요하지 않습니다.

- normalizer 코드 변경
- renderer 코드 변경
- fixture JSON 변경
- routing 결과 변경
- HWPX payload 변경
- HWPX output 재생성
- 실제 API, Make.com, Email 연동 구현

현재 필요한 작업은 저장 가능한 항목과 저장 금지 항목을 문서로 구분하는 것입니다.

## 로그 유형 구분

| 로그 유형 | 현재 상태 | 용도 |
|---|---|---|
| 반복 운영 로그 | 사용 가능 | placeholder 기반 로컬 PoC 반복 실행 결과 기록 |
| 사용자 preview 로그 | 문서 기준으로만 유지 | 한컴 수동 검수 결과와 중단 사유 기록 |
| 상태별 중단 로그 | 문서 기준으로만 유지 | `blocked`, `needs_security_review`, `template_required` 등 중단 상태 기록 |
| 외부 연동 검토 로그 | 구현 보류 | 향후 API, Make.com, Email 연동 검토 시 승인ㆍ보류ㆍ중단 사유 기록 |
| 외부 서비스 실행 로그 | 작성 금지 | 실제 외부 요청ㆍ응답, 실제 발송, 실제 계정 연결을 전제로 하므로 현재 범위 밖 |

## 반복 운영 로그와 외부 연동 검토 로그의 공유 항목

다음 항목은 실제값을 포함하지 않는 경우에만 공통으로 기록할 수 있습니다.

| 항목 | 기록 형식 | 기준 |
|---|---|---|
| 실행일 | `[YYYY.MM.DD. 확인 필요]` | 실제 운영 일시는 조직 기준 확인 전 확정하지 않음 |
| 실행 목적 | `[반복 운영 목적 확인 필요]` | 실제 사건명, 실제 기관명, 실제 문서명 제외 |
| 문서 유형 | `one_page_report` 등 허용 document_type | 실제 문서 제목이 아니라 유형만 기록 |
| 입력 유형 | `placeholder fixture` / `비식별 사용자 입력` / `[확인 필요]` | 실제 원문 사용 여부를 함께 확인 |
| 보안등급 | `A` / `B` / `[확인 필요]` | C/D등급은 외부 처리 후보에서 제외 |
| routing 상태 | `ready_for_draft` 등 상태값 | 실제 본문 내용 없이 상태값만 기록 |
| renderer 상태 | `rendered` / `template_required` / `skipped` / `[확인 필요]` | output 내용은 기록하지 않음 |
| 사용자 preview 상태 | `통과` / `보류` / `실패` / `[사용자 확인 필요]` | 사용자가 직접 확인한 결과만 기록 |
| 중단 사유 | 정해진 상태 또는 증상 | 실제 원문ㆍ파일명ㆍ개인정보 없이 범주만 기록 |
| 다음 조치 | `재시도` / `보류` / `사람 검토` / `[확인 필요]` | 자동 발송 또는 자동 승인으로 쓰지 않음 |

## 저장 허용 항목

다음 항목은 placeholder 기반이거나 실제값을 포함하지 않는 경우에만 저장할 수 있습니다.

- 문서 유형 코드
- routing 상태값
- validation 통과 여부
- `remaining_placeholders` 개수
- `missing_fields` 개수
- `template_required` 여부
- output Git 제외 확인 결과
- 사용자 preview 통과ㆍ보류ㆍ실패 상태
- 오류 범주
- 체크리스트 통과 여부
- `[확인 필요]`, `[사용자 확인 필요]` 같은 검토용 표시

허용 항목이라도 실제 원문, 실제 이름, 실제 기관명, 실제 문서번호, 실제 파일명이 포함되면 저장하지 않습니다.

## 저장 금지 항목

다음 항목은 로그와 감사 추적에 저장하지 않습니다.

- 실제 공문 원문
- 실제 보고서 원문
- 실제 민원 내용
- 실제 개인정보
- 실제 기관명, 관서명, 업체명, 담당자명
- 실제 문서번호, 민원번호, 접수번호, 사건번호
- 실제 이메일 주소, 전화번호, 주소
- 실제 HWPX 원본 파일명
- 실제 내부 경로
- 실제 API 요청 본문
- 실제 API 응답 본문
- 실제 이메일 수신자, 참조자, 첨부 파일명
- API 키, 토큰, 인증정보, 환경변수 예시값
- 실제 예산액, 견적 금액, 계약 조건
- 실제 내부망 주소, 시스템명, 계정정보

## 사용자 승인 기록 기준

사용자 승인 또는 보류는 실제 이름 대신 역할과 상태로만 남깁니다.

| 항목 | 허용 형식 | 금지 |
|---|---|---|
| 확인 주체 | `[담당 역할 확인 필요]` | 실제 담당자명 |
| 확인 방식 | `한컴 수동 preview` / `문서 검토` | 실제 내부 결재 경로 |
| 승인 상태 | `통과` / `보류` / `반려` / `[사용자 확인 필요]` | 자동 승인 |
| 확인 메모 | 증상 범주와 조치 방향 | 실제 원문, 실제 파일명, 실제 기관명 |

승인 상태는 초안 생성 흐름의 검수 상태일 뿐, 실제 발송ㆍ결재ㆍ계약ㆍ예산 집행 승인이 아닙니다.

## 중단 상태 기록 기준

중단 상태는 실제값 없이 다음 형식으로 기록합니다.

| 상태 | 기록 기준 |
|---|---|
| `blocked` | 실제값 의심, 외부 처리 금지, 비식별 재작성 필요 |
| `needs_security_review` | 사람 보안 검토 전 외부 전송과 HWPX 렌더링 중단 |
| `template_required` | 로컬 placeholder 템플릿 없음, output 생성 시도 없음 |
| validation 실패 | payload 또는 placeholder map 확인 필요 |
| `remaining_placeholders` 있음 | 사용자 preview 중단, template 또는 map 확인 필요 |
| Git 제외 실패 | HWPX, output, 실제 원본이 Git Changes에 보이는지 확인 필요 |

중단 로그에는 실제 원문 일부를 복사하지 않고, 상태와 조치 방향만 기록합니다.

## 오류 기록 기준

오류는 재현에 필요한 최소 범주만 기록합니다.

| 허용 | 금지 |
|---|---|
| 명령 이름 | 실제 입력 전문 |
| exit code | 실제 원문이 포함된 콘솔 출력 전문 |
| 오류 범주 | 실제 파일명 또는 실제 내부 경로 |
| 영향 범위 | 실제 업무 사건명 |
| 다음 조치 | 실제 담당자명 또는 개인 연락처 |

콘솔 출력에 실제값이 섞일 가능성이 있으면 전문을 붙여넣지 않고 요약합니다.

## 외부 연동 검토 로그의 추가 항목

향후 외부 연동 검토가 승인되더라도 다음 항목까지만 먼저 문서 기준으로 검토합니다.

| 항목 | 기준 |
|---|---|
| 연동 후보 | `OpenAI API` / `Make.com` / `Email draft` 같은 후보명 |
| 연동 상태 | `검토 중` / `보류` / `승인 전` / `구현 금지` |
| 외부 전송 여부 | 기본값 `전송하지 않음` |
| 승인 필요 여부 | `필요` 또는 `[확인 필요]` |
| 테스트 계정 여부 | `[확인 필요]` |
| 테스트 데이터 여부 | placeholder 기반 여부만 기록 |
| 비용 영향 | `[확인 필요]` |

실제 요청 ID, 실제 응답 ID, 실제 계정명, 실제 수신자 정보는 현재 저장하지 않습니다.

## 보관 위치와 기간

현재 저장소에 남길 수 있는 것은 placeholder 기반 기준 문서와 비식별 요약뿐입니다.

| 항목 | 현재 기준 |
|---|---|
| 실제 운영 로그 | 저장소에 저장하지 않음 |
| 비식별 반복 운영 요약 | 필요 시 문서 기준으로만 저장 |
| HWPX output | Git 제외 유지 |
| local HWPX template | Git 제외 유지 |
| 로그 보관 기간 | `[확인 필요]`, 조직 정책 확인 전 임의 확정 금지 |
| 로그 접근 권한 | `[확인 필요]`, 실제 운영 전 별도 검토 필요 |

## 코드 변경 판단

이번 기준 정리는 문서 기준으로 충분합니다.

변경하지 않는 항목:

- `normalizers/`
- `renderers/`
- `examples/json/`
- routing fixture
- HWPX payload mapper
- HWPX renderer output
- local HWPX template

## 다음 작업 후보

다음 단계는 실제 연동 구현이 아니라 실제 원문 차단과 비식별 입력 확인 절차를 문서 기준으로 구체화하는 것입니다.

검토 질문:

- 실제 원문이 섞였는지 어떤 기준으로 중단할지
- 비식별 입력이 A/B등급 후보인지 어떻게 확인할지
- C/D등급 또는 애매한 입력을 어떻게 보류할지
- 외부 전송 전 사람이 확인해야 할 최소 항목은 무엇인지

## 보안 검수

- 실제 개인정보 추가 없음
- 실제 기관명, 업체명, 담당자명 추가 없음
- 실제 문서번호, 민원번호, 사건번호 추가 없음
- 실제 날짜, 예산, 실적 수치 추가 없음
- 실제 공문 원문 또는 보고서 원문 추가 없음
- 실제 HWPX 원본 파일 추가 없음
- API 키, 토큰, 인증정보 예시값 추가 없음
- Email, API, Make.com 연동 구현 없음

## 결론

로그와 감사 추적 기준은 문서 기준으로 구체화했습니다.

현재 단계에서는 실제 연동 로그를 생성하지 않고, 저장 가능한 placeholder 기반 메타데이터와 저장 금지 실제값을 구분하는 수준에서 멈춥니다.
